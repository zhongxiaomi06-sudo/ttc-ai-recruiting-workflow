方案四：AI 猎头工作流 · AI Native 异常驱动调度

AI 默认自动推进全部环节；只在遇到异常、信息缺失或必须真人动作时，才生成 HTML 任务页调度人解决

AI Native异常驱动人机调度 状态机编排HTML 任务页面 证据链飞书/ChatGPT 自动采集

1. 核心设计原则

原则一

AI 是项目经理

AI Orchestrator 持有每个招聘任务的完整状态，决定下一步该做什么：解析 JD、搜人、打分、生成话术、还是调度人。

原则二

人是被调度的工具

猎头、顾问、合规 reviewer 都是系统可调用的人力资源。系统只在置信度不足或必须真人动作时，才生成 HTML 任务卡片派给人。

原则三

输出即 HTML 文档

所有给人类阅读的内容——电话清单、候选人档案、审核任务、反馈表单——统一由 Daemon 渲染为 HTML 页面，而不是原始 JSON 或聊天消息。

1.5 AI Native 异常驱动：什么时候调度人

核心规则

AI 先跑，跑不通才叫人。 正常路径上所有解析、搜索、评分、生成话术都由 AI 自动完成；只有在置信度不足、调用失败、数据缺失或需要真人动作时，系统才暂停并生成 HTML 任务页。

3.1 Agent 编排方法：层级状态机，而非群聊

我们不采用“多个 Agent 在一个聊天室里自由讨论”的开放协作模式，因为那会导致不可追踪、不可重复、成本不可控。

本方案采用 层级状态机（Hierarchical State Machine）：

  1. Orchestrator 是唯一的决策中心，持有 Mission 状态。
  2. 每个子 Agent 是单一职责的函数/服务，被 Orchestrator 按状态调用。
  3. 状态转移由规则 + LLM 联合决定：硬规则保证安全边界，LLM 负责模糊判断。
  4. 所有动作、决策理由、证据写入 agent_runs 表，可审计。

复用成熟读取工具

TTC 不自己写通用抓取/文件解析。读取层优先复用：

  • 公开网页：Crawl4AI（本地、免费、Apache-2.0）或 Firecrawl（托管 API，即开即用）
  • 需要登录/交互的页面：Playwright 或 Browser Use（LLM 驱动的浏览器 agent）
  • PDF/Word/PPT/Excel：MarkItDown（Microsoft 开源）或 Apache Tika 兜底

配置即可切换，未配置时自动回退到内置 requests + Playwright。详见 部署说明。

LLM 提升准确率

当配置了 LLM（OpenAI / Kimi / Claude 等）时，artifact_classifier 会优先调用 LLM 判断内容类型；JD 解析也会由 LLM 提取岗位、技能、城市、经验等结构化字段。未配置时自动使用关键词兜底。

飞书 Bot 主动通知

新的人类任务（打电话、异常处理）生成后，系统可通过飞书群机器人自动推送卡片，猎头点击卡片即可进入 HTML 任务页，无需主动刷新 Dashboard。

4. 人类作为工具：HTML 任务页面

每个需要人参与的动作，都会被封装成一个 human_task。系统不会把人拉进对话，而是给出一份结构化的 HTML 任务单。

猎头打电话（Phone Caller）
触发条件：候选人 overall_score ≥ 阈值且已生成话术
HTML 页面包含：候选人档案 · JD 摘要 · 匹配度与证据 · 电话话术 · 一键反馈按钮

触发条件：多个高分候选冲突 / 客户 JD 理解不清 / 需要确认推荐策略

触发条件：来源不可访问 / 信息冲突 / 隐私或合规红灯

5. Mission 状态机：一个招聘任务的完整生命周期

每一个 JD 对应一个 Mission。Orchestrator 按状态机推进，状态持久化在 SQLite 中。

created
收到 JD，未解析
jd_parsed
JD 解析完成，结构化字段就绪
sourcing
正在人才库 / 公网 / Collector 召回
scored
评分排序完成
human_review
需要顾问审核（可选）
problem_pending
AI 遇到异常/缺失，等待人解决
calling
生成电话任务，派给猎头
human_pending
等待人类完成任务
