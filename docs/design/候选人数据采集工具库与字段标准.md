# 候选人数据采集工具库与字段标准

> 更新日期：2026-07-06  
> 目标：将公开或已授权的网页、PDF、Office 文档和移动端素材转换为可追溯、可去重、适合 AI 评估的候选人数据。

## 一、核心结论

没有任何一个工具能够合法、稳定地提取“任何网站”。登录权限、验证码、网站条款、访问控制、个人信息保护与页面变化都是硬边界。

不建议同时安装和维护全部 10 个项目。建议组合为：

- 主抓取层：Crawl4AI（自托管优先）或 Firecrawl（快速 API 化）。
- 交互页面：Browser Use，仅处理普通抓取无法完成的授权交互。
- 本地文件：MarkItDown，负责 PDF、Word、PPT、Excel 等文件的 Markdown 转换。
- 大规模任务：Python 团队选 Scrapy；TypeScript 团队选 Crawlee。
- 特殊兜底：Scrapling 处理结构频繁变化的公开页面。

## 三、工具选择决策

| 数据场景 | 首选 | 备选 |
|---|---|---|
| 普通公开网页 | Crawl4AI | Firecrawl |
| 需要 JavaScript 渲染 | Crawl4AI/Playwright | Browser Use |
| 已授权的多步点击流程 | Browser Use | 定制 Playwright |
| 数万级静态页或列表页 | Scrapy | Crawlee |
| TypeScript 生产级爬取编排 | Crawlee | Firecrawl |
| PDF、Word、PPT、Excel | MarkItDown | 原生文档解析器/OCR |
| 页面布局变化频繁 | Scrapling | Crawl4AI |
| 自有 Android 设备上的授权 App | scrcpy + ADB/自动化脚本 | 平台官方导出/API |

## 四、候选人数据标准

每一条结论必须同时保留“结构化值＋原文证据＋来源 URL＋采集时间”。不允许 AI 只输出无来源的综合描述。

### 4.1 来源字段

- `source_url`：原始链接或本地文件标识。
- `source_type`：简历、公开职业主页、公司官网、采访、演讲、文章或其他。
- `collected_at`：采集时间。
- `content_hash`：原始快照哈希，用于去重和版本追踪。
- `access_basis`：公开、本人提供、用户授权或官方 API。
- `evidence_text`：支持结论的最小原文片段。
- `confidence`：高/中/低，不得将推测标记为事实。

### 4.2 候选人基础字段

- 姓名、英文名、公开职业别名。
- 当前城市、意向城市、可能的搬迁性。
- 本科院校、学历层次、毕业年份。211/985 标记。
- 出生年或年龄只记录候选人明确公开/提供的信息。仅能从毕业年份推测时，记录为年龄区间并标注低置信度，不从照片推断年龄。
- 工作年限、当前职级、薪资与到岗周期。薪资等非公开信息必须由候选人或授权方提供。

### 4.3 履历和项目字段

- `consulting_experience`：咨询公司、消费行业项目、职级、是否直接面对客户。
- `inhouse_experience`：甲方公司、品牌/产品/渠道/战略职能。
- `consumer_sector`：食品饮料、美妆、零售、家居、服饰、宠物等。
- `project_problem`：项目要解决的真实问题。
- `personal_role`：主导、子模块负责、参与或协助。
- `method`：消费者访谈、定量研究、竞品对标、TAM、UE、品类规划等。
- `decision`：最终推动了什么产品、品牌、渠道或组织决策。
- `implementation`：是否参与上市、运营、跨部门协作和复盘。
- `business_result`：收入、市占率、GMV、毛利、转化率、交付效率等可验证结果。
- `client_seniority`：服务过的最高客户/汇报对象层级。
- `commercial_consulting`：是否做过售前、需求界定、报价、交付、续约或客户经营。

### 4.4 启承资本岗位标签

- `consulting_plus_inhouse`：是否同时具备正式咨询公司和甲方经历。
- `undergraduate_tier`：985 / 211 / 其他。
- `target_age_band`：公开 JD 与隐藏画像合并后建议使用 29–33 岁柔性区间，不建议将 31 岁设为精确硬门槛。
- `product_innovation`、`brand_growth`、`channel_strategy`：分别保留证据和实际结果。
- `founder_readiness`：是否有与创始人、CEO、事业部负责人对话的证据。
- `paid_advisory_readiness`：是否能将研究和策略包装为客户愿意付费的交付。

## 五、数据清洗与质量规则

1. 先保存原始快照，再转 Markdown，最后做字段抽取。
2. 公司名、职位名、院校名建立别名字典，但不覆盖原文。
3. 相同候选人通过姓名＋工作时间线＋学校做概率去重；不依赖单一姓名。
4. 冲突信息不自动覆盖，按“本人简历/官方来源 > 可验证公开主页 > 采访/会议介绍 > 第三方转载”排序。
5. “主导”“负责”“推动”必须与可验证的决策或结果绑定，否则标记为待面试验证。
6. 低置信度推测不参与自动淘汰，只生成追问。
7. 对每条记录保留删除、更正、过期和来源撤回机制。

## 六、推荐的采集流程

```text
公开/已授权来源
        ↓
访问分类（静态 / JS / 交互 / 文件 / Android）
        ↓
抓取原始快照＋来源元数据
        ↓
Markdown 清洗＋附件解析
        ↓
候选人 JSON 字段抽取
        ↓
实体去重＋时间线冲突检测
        ↓
岗位匹配评分＋证据链＋面试追问
        ↓
人工复核后进入人才库
```

## 七、启承资本人才搜索方向

核心搜索组合：

```text
(消费咨询 OR 品牌策略 OR 消费者洞察 OR 品类策略)
AND
(产品创新 OR 新品上市 OR GTM OR 渠道增长)
AND
(甲方 OR 品牌方 OR 零售平台)
```

优先履历：

- 2–4 年消费咨询＋2–4 年消费甲方。
- 985/211 本科，总经验约 7–9 年。
- 做过产品定义、品牌定位、品类创新、新市场进入或渠道增长。
- 能独立面对创始人/高管，并将策略转化为付费咨询交付。

## 八、安全与合规边界

- 只采集公开、候选人本人提供、用户已授权或官方 API 允许处理的数据。
- 不绕过登录、付费墙、验证码、访问控制或平台明确禁止的技术措施。
- 不采集与招聘无关的家庭、健康、证件、账户或私密社交数据。
- 手机号、邮箱等联系信息默认脱敏，仅对有业务需要的授权用户展示。
- 在实际上线前，应由专业人员根据采集地区、平台条款、保留周期和使用目的审核合规方案。

## 九、建议实施顺序

1. 先上 Crawl4AI＋MarkItDown，完成公开网页和简历文件的标准化。
2. 建立候选人 JSON 结构、证据链和去重规则。
3. 引入 Browser Use，仅处理少量已授权、必须交互的页面。
4. 数据量增大后，根据团队语言在 Scrapy 和 Crawlee 中二选一。
5. 只在明确场景下引入 Scrapling、scrcpy、AutoScraper 或 curl-impersonate，避免技术栈膨胀。
