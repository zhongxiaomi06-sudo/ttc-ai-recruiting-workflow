# TTC 招聘量化决策系统研发规划

## 1. 系统定位

需求中的“交易系统”不是金融交易软件，而是把猎头资源配置抽象为量化投资：

- 职位是标的。
- 顾问时间、触达额度和人才数据是资金。
- 推荐、面试、Offer、成交是收益。
- 客户反馈、需求变化和候选人响应是市场信号。
- 加仓、减仓和止损对应资源投入决策。

系统目标是在有限人力下，提高推荐量、面试量、Offer 量和最终成交量，同时把不可自动化的高价值判断交给顾问。

## 2. 两条需求的关系

### 主系统：Recruiting Quant OS

负责职位发现、投入价值判断、自动 Sourcing、任务派发、反馈追踪和仓位管理。

### 首个可交付能力：工程简历含金量评分

`需求2` 适合作为主系统的第一个独立模块。它不针对某个具体 JD，而是判断工程技术候选人的通用技术含金量，并给顾问一组进一步验证深度的问题。

先完成这个模块有三个价值：

1. 需求边界清楚，可以快速让 Wendy 和 York 验收。
2. 输出可直接进入候选人排序和 Sourcing 流程。
3. 能积累顾问修正和客户反馈，为后续模型学习提供标签。

## 3. 工程简历含金量评分 MVP

### 输入

- PDF、DOCX 或纯文本简历。
- 可选：公司、部门、项目链接和公开技术资料。

### 输出

- 总分 0-100，但必须同时给出置信度。
- 每个维度的分数、简历证据、扣分原因和未知项。
- “深但表达差”与“表达漂亮但证据浅”的区分。
- 建议追问的 5-10 个问题，以及每个问题要验证什么。
- 风险标签：包装过度、职责不清、指标不可验证、项目同质化、技术栈堆砌。

### 建议评分维度

| 维度 | 权重 | 判断内容 |
|---|---:|---|
| 技术深度 | 25 | 是否理解原理、边界、故障和权衡 |
| 项目所有权 | 15 | 是参与、主导还是独立负责 |
| 复杂度与规模 | 15 | 数据量、并发、延迟、可靠性、团队范围 |
| 结果与影响 | 15 | 是否有可验证指标和业务结果 |
| 工程完整性 | 10 | 测试、监控、部署、安全、维护 |
| 经历含金量 | 10 | 公司/部门/岗位环境，但证据不足时必须标未知 |
| 成长与连续性 | 5 | 职责是否升级、能力是否连续积累 |
| 证据可信度 | 5 | 描述是否具体、一致、可追问和可验证 |

不能仅凭公司名给高分。公司和部门知识必须带来源与更新时间；未知部门不得由模型猜测。

### 第一版技术方案

1. 文档解析：提取经历、项目、技能、指标和时间线。
2. 证据抽取：每个评分结论绑定简历原句。
3. 规则评分：先完成可解释的维度基线。
4. LLM 评审：分析深浅、所有权、技术权衡和追问点。
5. 一致性检查：时间冲突、指标异常、项目描述模板化。
6. 人工反馈：顾问修改分数和记录客户评价。

验收不以“模型觉得合理”为准，而以同一批简历上顾问评分一致性和客户反馈相关性为准。

## 4. Recruiting Quant OS MVP

### 核心状态机

```text
DISCOVERED 发现职位
    -> ASSESSED 完成评分
    -> TRIAL 小仓试投
    -> ACTIVE 正常推进
    -> SCALE_UP 加仓
    -> SCALE_DOWN 减仓
    -> STOP 止损/关闭
```

所有状态迁移必须保留触发信号、规则版本、操作者和时间，允许人工覆盖和事后回放。

### 核心模块

1. **职位监听**：CRM/ATS、客户群、顾问录入和公开渠道形成统一职位事件。
2. **职位理解**：结构化职责、硬要求、隐性要求、薪资、地点和目标公司。
3. **投入评分**：客户真实性、需求清晰度、人才供给、反馈速度、成功费和历史转化。
4. **仓位管理**：根据分数分配顾问时间、候选人数和触达预算。
5. **闪电战 Sourcing**：内部人才库优先，生成首批候选人并排序去重。
6. **任务派发**：机器无法完成时，生成有截止时间和验收标准的人类任务。
7. **反馈追踪**：记录推荐、回复、面试、Offer、拒绝和需求变化。
8. **策略复盘**：回放每次加减仓决策，比较实际收益，调整规则。

## 5. 第一版决策策略

第一版使用可解释规则，不直接训练黑箱模型。

```text
职位投入分 =
  需求清晰度 * 0.20
  + 客户反馈速度 * 0.20
  + 人才供给度 * 0.15
  + 历史转化率 * 0.20
  + 收益预期 * 0.15
  + 数据完整度 * 0.10
```

- 高分：进入闪电战并提高候选人额度。
- 中分：小仓试投，等待首轮反馈。
- 低分：只派发澄清任务，不自动扩大投入。
- 连续无反馈、需求反复变化或职位关闭：自动减仓/止损。

权重必须配置化，任何自动决策都展示原因，顾问可以覆盖。

## 6. 核心数据对象

- `JobOpportunity`：职位、客户、要求和状态。
- `SignalEvent`：职位新增、反馈、面试、需求变化等事件。
- `JobScore`：各维度分数、证据和规则版本。
- `PositionAllocation`：当前仓位、预算和变更理由。
- `CandidateProfile`：结构化候选人和通用含金量评分。
- `Recommendation`：候选人与职位的匹配及推进状态。
- `HumanTask`：需要顾问完成的动作、截止时间和结果。
- `Outcome`：面试、Offer、成交、拒绝和原因。

## 7. 核心指标

### 效率

- 新职位到首批候选人的时间。
- 每顾问小时产出的有效推荐数。
- 自动完成任务比例与人工接管原因。

### 质量

- Top-K 候选人被顾问接受率。
- 推荐到面试率、面试到 Offer 率。
- 简历含金量评分与客户评价的一致性。

### 经营

- 每个职位投入的人时与预期收益。
- 客户反馈 SLA。
- 加仓、减仓、止损决策后的真实转化变化。

## 8. 实施顺序

### 比赛期间

- 只保留需求澄清、数据字段确认和样本准备。
- 不启动大规模平台开发，不接外部平台自动化。

### 赛后阶段 1：简历评分器

1. 与 Wendy 收集 20-50 份脱敏工程简历及人工排序。
2. 定义评分 Rubric 和“深/浅/solid”的可观察证据。
3. 实现简历解析、证据评分、追问生成和反馈界面。
4. 做盲测并根据分歧修改 Rubric。

### 赛后阶段 2：职位评分与仓位

1. 统一职位、客户、反馈和任务数据模型。
2. 实现职位评分卡和可配置规则。
3. 实现 TRIAL/ACTIVE/SCALE_UP/SCALE_DOWN/STOP 状态机。
4. 用历史职位进行离线回放，不自动触达真实候选人。

### 赛后阶段 3：闪电战试点

1. 只选 3-5 个标准化工程职位。
2. 先接 TTC 内部人才库，生成候选人列表。
3. 顾问确认后再触达，记录所有修正和结果。
4. 达到验收指标后再扩大职位与数据源。

## 9. 安全与业务边界

- 候选人数据必须最小化、脱敏、分级授权并可删除。
- 不绕过 Boss、脉脉等平台规则抓取或批量自动联系。
- 自动评分不得成为唯一淘汰依据，必须允许人工复核。
- 公司/部门含金量判断必须有证据，禁止模型编造。
- “多账号自动发小红书、封号即弃”的需求不进入研发范围，存在平台滥用和账号安全风险。
- 所有自动推荐、加减仓和止损决策必须可解释、可回放、可覆盖。

## 10. 当前结论

TTC 项目不应一开始就开发完整量化平台。最合理顺序是：

> 工程简历含金量评分器 -> 职位投入评分 -> 仓位状态机 -> 闪电战 Sourcing -> 全流程试点。

这条路线既能快速交付 `需求2`，也能让每一步产生的数据成为 Recruiting Quant OS 的训练与决策基础。
